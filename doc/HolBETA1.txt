================================================================================
Hol BETA1 â€” Language Specification (Draft)
================================================================================

This document specifies the syntax and semantics of `Hol BETA1`, a dialect of
خ»Prolog that supersedes `Hol ALPHA2`. BETA1 keeps every program accepted by
ALPHA2 working and additionally provides:

  (A) Original خ»Prolog-style user-defined notation
      (`infixl`/`infixr`/`infix`/`prefix` with precedences),
      with runtime pretty-printing that preserves the user's notation.
  (B) An abbreviation mechanism, modelled after Coq's `Notation`/abbreviation
      facility (e.g. `string` as a display-time abbreviation for `list char`).
  (C) A `presburger : string -> o` predicate that parses its argument as a
      Presburger formula and decides entailment from the arithmetic constraints
      accumulated by the current proof branch. The solver is the one in
      `src/Calc/Presburger/Internal.hs`.
  (D) (Open / experimental) A hybrid representation that stores de Bruijn
      indices internally while retaining user-supplied binder names, أ  la Coq.
  (E) (Open / experimental) Interactive instantiation of flexible (logic)
      variables during a debugging session â€” the foundational primitive for
      tactic-style proof construction in the future `LoL` project.

Compatibility minimum: the test scripts

    einstein.sh   ( Example/einstein.hol )
    fi.sh         ( Example/fibonacci.hol )
    lbeta.sh      ( Example/test.hol )

must continue to pass with the same answer set.


================================================================================
Chapter 0. Overview
================================================================================

0.1 Design Goals
--------------------------------------------------------------------------------

BETA1 simultaneously pursues three goals.

  1. *Expressiveness* â€” higher-order Horn clauses with first-class arithmetic
     reasoning available on demand via the `presburger` predicate.
  2. *Surface fidelity* â€” accept the original خ»Prolog/Teyjus surface syntax,
     in particular fixity declarations and precedence levels, and replay them
     verbatim in answer output.
  3. *Forward compatibility with LoL* â€” design choices that make this
     interpreter a usable host for an interactive proof assistant kernel
     (named binders kept for display; debugger as proto-tactic engine).


0.2 Differences from ALPHA2
--------------------------------------------------------------------------------

  +-----------------------+-------------------------+--------------------------+
  | Aspect                | ALPHA2                  | BETA1                    |
  +-----------------------+-------------------------+--------------------------+
  | Notation declaration  | none                    | infixl/infixr/infix/prefix
  | Answer pretty-print   | built-in fixity table   | user notation preserved  |
  | Abbreviation          | none                    | `abbrev`/`notation`      |
  | Arithmetic            | collected only          | + `presburger : str->o`  |
  | Inconsistency pruning | n/a                     | implicit `presburger âٹ¥`  |
  | Binder representation | de Bruijn               | de Bruijn + name hints   |
  | Interactive debug     | step / continue         | + `:assign ?V := t`      |
  +-----------------------+-------------------------+--------------------------+


0.3 Notational Conventions
--------------------------------------------------------------------------------

Throughout this document we use the following meta-variables.

  - `c`, `d`     : data constants (lowercase head)
  - `X`, `Y`     : logic variables (uppercase head)
  - `t`, `u`     : terms
  - `A`, `B`     : types
  - `خ؛`          : kinds
  - `خ“`          : type / kind environments
  - `خ”`          : the program (a set of facts/clauses)
  - `خ¨`          : the set of hypotheses introduced by `=>`
  - `C`          : a multiset of constraints attached to a branch
  - `خ¸`          : a substitution
  - `د†`          : a Presburger formula

Chapter 1. Syntax

1.1 Lexical Structure
--------------------------------------------------------------------------------

Token classes:

  - lowercase ident : `[a-z][A-Za-z0-9_]*`     â€” constants, predicate names
  - uppercase ident : `[A-Z_][A-Za-z0-9_]*`    â€” logic variables
  - natural literal : `0 | [1-9][0-9]*`
  - character lit   : `'c'` (with the usual escapes)
  - string lit      : `"..."`
  - punctuation     : `( ) [ ] { } , . | \ :- => ;`
  - reserved words  : `kind`, `type`, `module`,
                      `infixl`, `infixr`, `infix`, `prefix`,
                      `abbrev`, `notation`,
                      `pi`, `sigma`, `is`, `true`, `fail`, `!`, `debug`,
                      `presburger`
  - line comment    : `% ...`
  - block comment   : `/* ... */`        (nested â€” see Q1)

There is no layout rule; every declaration and clause is terminated by `.`.


1.2 Kinds and Types
--------------------------------------------------------------------------------

  kinds  خ؛ ::= type | خ؛ -> خ؛
  types  A ::= a            -- type constant (introduced by `kind`)
             | A -> A
             | A B          -- type application
             | X            -- type variable (implicitly generalised)

Built-in type constants:

  o      : type           -- the type of goals/propositions
  nat    : type           -- natural numbers
  char   : type
  string : type           -- abbreviation for `list char` (see آ§1.6)
  list   : type -> type
  (->)   : type -> type -> type

Declarations:

  kind <lowercase> <خ؛>.
  type <lowercase> <A>.

Polymorphism is Hindleyâ€“Milner with implicit prenex generalisation at type
declarations. Type variables appearing free in a `type` declaration are
universally quantified at the outermost level.


1.3 Terms
--------------------------------------------------------------------------------

  t ::= c                   -- data constant
      | X                   -- logic variable
      | t t                 -- application
      | x \ t               -- خ»-abstraction (`x` is a name hint)
      | <literal>
      | [ t, ..., t ]       -- list sugar
      | t :: t              -- cons
      | []                  -- nil

Internally each `x \ t` is stored as a de Bruijn index together with the
name hint `x`. The name hint is used only for display and for diagnostic
messages; خ±-equivalence is decided on the de Bruijn form. See آ§3.3 for the
hybrid representation and آ§D (Open Questions) for issues around capture-free
display.


1.4 Facts and Clauses
--------------------------------------------------------------------------------

A program is an ordered list of declarations and clauses. A clause is one of

  <atom>.                                   -- a unit fact
  <atom> :- <body>.                         -- a Horn clause

The body grammar:

  body ::= <atom>
         | body , body                       -- conjunction
         | body ; body                       -- disjunction
         | body => body                      -- hypothetical implication
         | pi    x \ body                    -- universal
         | sigma x \ body                    -- existential
         | true | fail | !
         | t is <arith>                      -- arithmetic evaluation
         | presburger <string-literal>       -- Presburger entailment query
         | debug "..."

`=>`, `pi`, and `sigma` form the core of خ»Prolog's intuitionistic
proof-search calculus (see آ§2.2).


1.5 User-Defined Notation
--------------------------------------------------------------------------------

BETA1 supports the original خ»Prolog fixity declarations:

  infixl  <op>  <precedence>.
  infixr  <op>  <precedence>.
  infix   <op>  <precedence>.     -- non-associative
  prefix  <op>  <precedence>.

Rules:

  - Precedence is a natural number; *higher* binds *tighter* (matching the
    table currently used by `Hol/ALPHA2/TermNode.hs::checkOper`).
  - Operator tokens are either symbolic (`[!@#$%^&*+\-=<>?/|~:]+`) or a
    backtick-quoted lowercase identifier (TBD; see Q2).
  - Mixing different associativities at the same precedence is a parse
    error.
  - Declarations are one-pass and take effect on the lines that *follow* them.

The built-in fixity table (the BETA1 defaults, user-redefinable):

  prefix  Lambda 0
  infixr  :-     0
  infixl  ;      1
  infixr  =>     2
  infixl  ,      3
  infixr  ->     4
  infixr  ::     4
  prefix  pi     5
  prefix  sigma  5
  infix   =      5
  infix   =<     5
  infix   <      5
  infix   >=     5
  infix   >      5
  infix   is     5
  infix   +      6
  infix   -      6
  infix   *      7
  infix   /      7

This table is identical to `checkOper` in `src/Hol/ALPHA2/TermNode.hs`; the
difference is that BETA1 carries it as data (`NotationEnv`) rather than
hard-coded in the pretty printer.


1.6 Abbreviations
--------------------------------------------------------------------------------

BETA1 introduces a Coq-style abbreviation mechanism. Two forms are provided.

  (a) Type-level:

        abbrev <name> := <type>.

      Example:
        abbrev string := list char.

  (b) Term-level (a constant-headed pattern that elaborates to a term):

        notation <name> <pat-args> := <term>.

      Example:
        notation single X := X :: [].

Semantics:

  - At *parse time*, an occurrence of the abbreviation is expanded to its
    right-hand side. The resulting term/type is what the elaborator sees;
    abbreviations are *not* part of the kernel.
  - At *display time*, the pretty printer attempts to fold the right-hand
    side back to the abbreviation (most-specific-first). This is the
    distinguishing feature: `list char` is shown as `string` rather than
    as `list char`.
  - Folding is purely syntactic and stops at the first match; users can
    introduce ambiguous folds, and the order of declaration breaks ties.

Built-in abbreviations seeded at startup:

  abbrev string := list char.


1.7 Modules
--------------------------------------------------------------------------------

One file is one module. The module name is the file path with `/` replaced
by `.` (unchanged from ALPHA2). An optional `module <name>.` header may
appear as the first declaration. A proper `import` system is *not* part of
BETA1 (see Q5).

2.1 Term Rewriting
--------------------------------------------------------------------------------

The internal term representation (`TermNode`) uses de Bruijn indices with
explicit suspensions (the same machinery as `Hol/ALPHA2/TermNode.hs`).
Each `NLam` additionally carries a `NameHint :: Maybe SmallId`, set by the
desugarer and consumed by the pretty printer; it is ignored by reduction
and unification.

Reduction modes:

  - `WHNF` : weak head normal form
  - `HNF`  : head normal form
  - `NF`   : full normal form

خ²-reduction: `(خ».t) u âں¶ t[u/0]`, threaded through suspensions for laziness.
خ· is *not* applied eagerly; it is only considered inside HOPU.

2.3 Arithmetic Constraints and the `presburger` Predicate
--------------------------------------------------------------------------------

A branch's constraint multiset C is partitioned into:

  Constraint ::= DisagreementConstraint  âں¨t, uâں©    -- HOPU residual
              | EvaluationConstraint    âں¨t, uâں©    -- from `t is u`
              | ArithmeticConstraint    د†          -- Presburger atom

ALPHA2 already accumulates `EvaluationConstraint` and a degenerate form of
`ArithmeticConstraint`. BETA1 *makes them queryable*.

  presburger : string -> o.

Operationally:

  1. The string argument must reduce to a ground string literal.
     Otherwise the goal fails with a `bad-presburger` runtime error.
  2. The literal is parsed as a Presburger formula د† using the grammar of
     `Calc.Presburger.Internal.PresburgerTermRep` /
     `PresburgerFormula PresburgerTermRep`.
  3. Let خ¦ be the conjunction of all `ArithmeticConstraint`s currently in C.
     The goal succeeds iff

         خ¦  âٹ¨  د†        (i.e.  خ¦ âˆ§ آ¬د†  is unsatisfiable),

     decided by the procedure exported from
     `src/Calc/Presburger/Internal.hs`.
  4. The check produces no substitution and adds no new constraints.

Implicit consistency pruning. At every BACK / AND step, BETA1 conceptually
inserts the goal

        presburger "_|_", !, fail

before evaluating the original goal. In words: "if âٹ¥ is entailed by the
current خ¦, then commit and fail this branch". This recovers the eager
consistency-check behaviour suggested in earlier drafts, but as a derived
notion expressed in terms of the user-visible predicate.

Implementations are free to omit the implicit check when the constraint
store has not changed since the previous check; equivalence is up to branch
pruning behaviour, not to answer correctness.

Examples.

  ?- X is 3, presburger "X > 5".
  no.

  ?- X is 3, presburger "X = 3".
  yes.

  ?- presburger "exists Y. Y + Y = X", X is 4.
  yes.

  ?- X > 5, X < 3.        % implicitly `presburger "_|_"` succeeds â†’ fail
  no.


2.4 The `is` Predicate
--------------------------------------------------------------------------------

`tâ‚پ is tâ‚‚` is evaluated as follows.

  - Normalise `tâ‚‚` to NF.
  - If NF(`tâ‚‚`) is a closed arithmetic term, evaluate it to a natural-number
    literal `n` and unify `tâ‚پ` with `n`.
  - Otherwise, add `EvaluationConstraint tâ‚پ tâ‚‚` to C.
  - Whenever both sides of a pending `EvaluationConstraint` become closed
    arithmetic terms, it is promoted to an `ArithmeticConstraint` of the
    form `EqnF tâ‚پ tâ‚‚`.

This is the same behaviour as ALPHA2; the only change is that the resulting
`ArithmeticConstraint`s are now observable to user code via `presburger`.


2.5 Answer Substitution
--------------------------------------------------------------------------------

When the proof search reaches success on a query Q, the interpreter:

  1. Restricts the substitution to the free variables of Q and their
     transitive dependencies (as ALPHA2 already does).
  2. Reports the residual constraint set, *with the syntactic concession*
     that any `ArithmeticConstraint` whose negation is implied by خ¦ is
     dropped (it is trivially true and uninformative).
  3. Pretty-prints the bindings using the rules of آ§2.6 and آ§2.7.


2.6 Notation-Preserving Pretty Printing
--------------------------------------------------------------------------------

Principle. Any operator the user introduced with `infixl`/`infixr`/`infix`/
`prefix` is rendered in that form, at that precedence, on output. The
printer must never silently expose internal de Bruijn indices, `NApp`
chains, or built-in head symbols where a user-facing notation exists.

Mechanism.

  - The parser attaches `Maybe (Fixity, Precedence)` to every data constant
    it introduces.
  - `constructViewer` (currently in `Hol/ALPHA2/TermNode.hs`) is rewritten
    to consult a `NotationEnv :: Map Constant (Fixity, Precedence)` carried
    alongside `KindEnv`/`TypeEnv`, instead of the hard-coded `checkOper`.
  - Built-ins are seeded into `NotationEnv` at startup (the table of آ§1.5).

The public `Show TermNode` instance keeps its signature but is now defined
in terms of the configured `NotationEnv`.


2.7 Abbreviation Folding
--------------------------------------------------------------------------------

Display-time folding is a separate pass that runs *after* notation pretty-
printing has produced a `ViewNode`. For each registered abbreviation

        abbrev S := E

the viewer tries to rewrite occurrences of (the view of) `E` back to `S`,
most-specific-first. Folding is purely textual on the `ViewNode` AST; it
never changes the underlying term.

Term-level `notation` definitions fold analogously: their right-hand sides
are interpreted as `ViewNode` patterns with the declared formal parameters
acting as metavariables.


2.8 Names vs. de Bruijn (Hybrid Representation)
--------------------------------------------------------------------------------

For binders, BETA1 attaches an optional name hint to each `NLam`:

  data TermNode
      = ...
      | NLam { _hint :: Maybe SmallId, _body :: TermNode }
      | ...

Hints flow through the desugarer and through suspensions, but they do not
affect خ±-equivalence, unification, or reduction. They serve three purposes.

  1. *Pretty printing*: instead of `W_3\ W_2\ ...`, the viewer can render
     `X\ Y\ ...` whenever the original hints are non-clashing and
     non-shadowing. When a clash would arise, the viewer suffixes a digit.
  2. *Diagnostics*: HOPU disagreement reports cite the original names where
     possible.
  3. *Future tactic UI* (`LoL` integration): a debugger session that asks
     "instantiate ?V to what?" can present the variable in the user's own
     vocabulary.

Hints are deliberately *advisory* â€” the runtime is correct iff it would be
correct after erasing every `_hint`.

Chapter 3. Runtime and Debugger
================================================================================

3.1 REPL Commands
--------------------------------------------------------------------------------

Same as ALPHA2 plus the following additions.

  :d                 toggle debug mode (unchanged)
  :assign ?V := t.   instantiate a flexible variable to a user-supplied term
                     at the current breakpoint (debug mode only)
  :show ?V           show the current value (or "unbound") of ?V
  :constraints       dump the current constraint multiset C
  :q                 quit (unchanged)

`:assign` is the seed of an interactive tactic engine. Its preconditions:

  - The session is in debug mode and at a step boundary.
  - The named variable is currently flexible (a `LVar`).
  - The supplied term type-checks against ?V's expected type.

On success, the assignment is committed as a normal HOPU substitution; on
failure, the debugger reports the error and stays at the same breakpoint.


3.2 Debug Output
--------------------------------------------------------------------------------

In debug mode the interpreter prints, before each transition:

  - the goal under consideration,
  - the constraint multiset C (with arithmetic constraints rendered in
    `Calc.Presburger` form),
  - the current `NotationEnv` and abbreviation table (compact form),
  - the binding substitution restricted to user-visible names.

4.2 Presburger Bridge
--------------------------------------------------------------------------------

`Arith.hs` exposes:

  parsePresburger :: String -> Either ErrMsg MyPresburgerFormulaRep
  entails         :: [MyPresburgerFormula] -> MyPresburgerFormula -> Bool

`entails خ¦ د†` is implemented as the unsatisfiability check on
`خ¦ âˆ§ آ¬د†` provided by `Calc.Presburger.Internal`. BETA1 makes no assumption
about the solver's internals beyond decidability.

Lifting of `ArithmeticConstraint`s currently in C to `MyPresburgerFormula`
is best-effort: a term outside the linear-integer fragment (e.g. a product
of two variables) is *omitted* from خ¦. This is sound for entailment in the
direction we use it: dropping conjuncts from خ¦ can only make fewer formulas
entailed, never more, so `entails` never erroneously reports success.


4.3 Named-Binder Display
--------------------------------------------------------------------------------

The hybrid representation has one subtle invariant: the name hint must
either be (i) unique among enclosing binders or (ii) absent. Whenever the
desugarer introduces a binder whose hint would shadow an enclosing one, it
either silently خ±-renames the *fresh* binder's hint (preferred) or drops it
to `Nothing`. The runtime never has to consult hints for correctness, so a
missing hint is always safe.


4.4 Test Compatibility
--------------------------------------------------------------------------------

  - For every file in `Example/*.hol` the answer set produced by BETA1 must
    be identical to the answer set produced by ALPHA2. Differences in
    pretty-printed form are tolerated only insofar as they reflect the
    user's notation/abbreviation declarations (آ§2.6, آ§2.7).
  - `einstein.sh`, `fi.sh`, `lbeta.sh` must exit 0 with byte-identical
    answer strings modulo whitespace and notation folding.
