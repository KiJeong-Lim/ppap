# Project A: Go-to-Extraction Differential Fuzzing

## 0. Goal

Project A checks whether a generated Go program behaves the same through two
execution paths:

```text
Path A:
  gofile.go
    -- Go compiler -->
  native executable
    -- run -->
  observable behavior

Path B:
  gofile.go
    -- translator -->
  gofile.v
    -- Coq compilation + extraction -->
  extracted Haskell
    -- GHC/run -->
  observable behavior
```

The core property is:

```text
obs(run(go build gofile.go))
==
obs(run(extracted Haskell))
```

The Go compiler and runtime are the oracle. The translator, generated Coq,
extraction setup, and Haskell runtime bridge are the implementation under test.

The initial target is not full Go semantics. It is end-to-end differential
fuzzing for a small, well-typed Go subset. The generator, shrinker, feature
coverage, corpus, and scoring loop can grow after the runner is stable.

## 1. Observable Behavior

The MVP compares only stable observations:

```text
termination class
exit code
stdout bytes
timeout flag
```

Suggested shape:

```hs
data Obs = Obs
  { obsTermination :: Termination
  , obsExitCode    :: Maybe Int
  , obsStdout      :: ByteString
  , obsTimedOut    :: Bool
  }

data Termination
  = Terminated
  | TimedOut
  | RuntimeFailed
```

Initial comparison policy:

```text
1. If both paths terminate normally, compare exit code and stdout bytes.
2. If native Go succeeds and the extracted path fails, report a mismatch.
3. If native Go fails and the extracted path succeeds, report a mismatch.
4. If both paths time out, classify the case as inconclusive.
5. If only one path times out, report a mismatch.
```

Stderr is stored as a log but is not part of equality. Runtime messages often
contain implementation-specific wording.

## 2. Failure Classification

Failures should be classified by pipeline stage instead of being collapsed into
one generic failure.

```hs
data PipelineResult
  = InvalidGo GoBuildLog
  | TranslatorError Log
  | CoqError Log
  | ExtractionError Log
  | HaskellCompileError Log
  | NativeRunError Obs
  | ExtractedRunError Obs
  | RanBoth Obs Obs

data Failure
  = TranslatorCompletenessBug
  | IllTypedGeneratedCoq
  | ExtractionSetupBug
  | HaskellRuntimeIntegrationBug
  | ObservableMismatch Obs Obs
  | TerminationMismatch Obs Obs
```

Classification policy:

```text
Go rejected the source:
  The generator produced invalid Go. Discard it as a generator bug.

Go accepted, translator failed:
  Candidate translator completeness bug.

Go accepted, translator succeeded, Coq compilation failed:
  Candidate generated-Coq typing or soundness bug.

Coq compilation succeeded, extraction failed:
  Candidate extraction setup bug.

Extraction succeeded, GHC failed:
  Candidate generated-Haskell or runtime bridge bug.

Both paths ran, observations differ:
  Candidate semantic mismatch.

Both paths timed out:
  Inconclusive until the timeout or program is adjusted.
```

The most important early failures are `CoqError` and `ObservableMismatch`.

## 5. Phase 0: Deterministic Runner

The first implementation should run one seed from start to finish. Do not start
with QuickCheck-style search.

Tasks:

```text
1. Generate a small Go program from a seed.
2. Run gofmt.
3. Build the native Go executable.
4. Run the native executable with a timeout.
5. Run the translator to produce gofile.v.
6. Compile the Coq file and extraction harness.
7. Check that extracted Haskell exists.
8. Compile and run the extracted Haskell path.
9. Compare the two observations.
10. Save every artifact and result.json.
```

Acceptance:

```text
Project one --seed=1 --size=0
```

This should create one work case and record both observations. At this phase,
the generator may produce only:

```go
package main

import "fmt"

func main() {
    fmt.Println(3)
}
```

## 6. Phase 1: Small Typed Go AST

Generate a typed AST in Haskell, then pretty-print it to Go. Do not generate
random source text directly.

Initial subset:

```text
types:
  int
  bool
  string

expressions:
  literals
  variables
  +, -, *, /, %
  ==, !=, <, <=, >, >=
  &&, ||, !

statements:
  var declaration
  assignment
  block
  if / else
  bounded for
  fmt.Println
```

Initial exclusions:

```text
goroutines
channels
map iteration
reflection
unsafe
floating point
panic / recover
defer
interfaces
methods
filesystem / network / time / randomness
unbounded loops
```

Acceptance:

```text
1. gofmt accepts all generated sources.
2. go build rejects almost no generated cases.
3. Rejected Go is recorded as a generator bug, not an equivalence failure.
```

## 7. Phase 2: Boolean Differential Fuzzing

Before score or corpus search, run simple Boolean differential checks:

```hs
prop_equiv :: TestCase -> IO PropertyResult
prop_equiv tc = do
  result <- runPipeline tc
  case result of
    InvalidGo _ ->
      pure Discard

    TranslatorError _ ->
      pure (Fail TranslatorCompletenessBug)

    CoqError _ ->
      pure (Fail IllTypedGeneratedCoq)

    ExtractionError _ ->
      pure (Fail ExtractionSetupBug)

    HaskellCompileError _ ->
      pure (Fail HaskellRuntimeIntegrationBug)

    RanBoth obsGo obsHs ->
      if comparable obsGo obsHs && obsGo == obsHs
      then pure Pass
      else pure (Fail (ObservableMismatch obsGo obsHs))
```

Acceptance:

```text
1. Fixed seeds produce fixed cases and fixed results.
2. Every failure leaves artifacts.
3. replay reproduces a saved failure.
```

## 8. Phase 3: AST Shrinking

Shrinking should operate on the Go AST, not on rendered source text.

Shrink rules:

```text
program:
  remove statements
  replace a block with its contents
  remove unused variables

statement:
  replace if with then or else branch
  remove else branch
  reduce loop bound
  shrink loop body
  shrink assignment RHS
  reduce println arguments

expression:
  replace binary expression with left or right subexpression
  shrink literals to 0, 1, -1, "", "a", false, true
  simplify x + 0, x * 1, and similar expressions
  shrink division/modulo divisors to small nonzero values
```

Shrinking loop:

```text
1. Start from a failing TestCase.
2. Generate candidate shrinks.
3. Run the pipeline on each candidate.
4. Accept a candidate if the same failure class is preserved.
5. Stop when no smaller candidate preserves the failure.
6. Save shrunk/gofile.go and a report.
```

## 10. Phase 5: Score

After Boolean fuzzing is stable, add a score objective:

```text
Boolean property:
  Is this a counterexample?

Score objective:
  How close or interesting is this passing case?
```

Suggested shape:

```hs
data Score
  = FoundCounterexample Failure
  | Interesting Double
  | Irrelevant
```

Initial risk signals:

```text
division/modulo near boundary values
negative operands in division/modulo
short-circuit expressions
conditions near true/false boundaries
loop bound 0 or 1
integer overflow boundaries
variable shadowing
zero initialization
```

Acceptance:

```text
1. Passing cases can still receive scores.
2. Low-score cases can be kept in a corpus.
3. Score and feature coverage appear in reports.
```

## 11. Phase 6: Corpus-Guided Mutation

After score exists, keep a corpus.

Corpus admission criteria:

```text
1. New feature appears.
2. Existing feature combination appears in a smaller program.
3. Risk score is low.
4. Timeout or runtime boundary is approached.
5. A shrunk regression seed is discovered.
```

Mutation examples:

```text
replace expression
insert statement
delete statement
duplicate block
toggle boolean literal
perturb integer literal
change loop bound
wrap statement in if
wrap statement in bounded for
rename variable to create shadowing
```

## 12. Extraction Performance

Compiling Coq and Haskell for every case is slow, but correctness comes first.
Optimize only after the end-to-end path is reliable.

Optimization order:

```text
1. Cache fixed Coq libraries.
2. Compile only the generated Coq file per case.
3. Separate the fixed Haskell runtime driver.
4. Add case-level parallelism.
5. Add result caching keyed by input and tool versions.
```

## 13. Timeout Policy

Every external process must have a timeout.

Initial defaults:

```text
gofmt: 1s
go build: 2s
native run: 1s
translator: 2s
coqc: 5s
ghc: 5s
extracted run: 1s
```

Timeouts are observations, not ordinary crashes.

## 15. Regression Policy

Every confirmed failure should become a regression after shrinking.

Suggested layout:

```text
regressions/
  issue-0001-short-circuit-side-effect/
    gofile.go
    input.stdin
    expected.json
    report.txt
```

`expected.json` should describe the correct behavior after the bug is fixed. If
the bug is still open, mark the regression as known failing.

## 18. Immediate Next Actions

Recommended order:

```text
1. Keep the deterministic runner green.
2. Keep the translator binary contract stable.
3. Make stdout behavior exact for generated fmt.Println cases.
4. Expand the generator one feature at a time.
5. Add targeted regressions as soon as a mismatch is found.
6. Add score/corpus only after replay and shrink are reliable.
```

## 19. Guiding Principles

```text
1. Go compiler/runtime is the oracle.
2. The Coq/extraction path is the implementation under test.
3. Generate well-typed AST, not random text.
4. Keep every failure reproducible.
5. Shrinking is not optional.
6. Start with Boolean differential fuzzing.
7. Add scoring only after the pipeline is stable.
8. Expand Go features one semantic risk at a time.
```
